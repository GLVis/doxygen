15 # How to Contribute
16 
17 The GLVis team welcomes contributions at all levels: bugfixes; code improvements;
18 simplifications; new visualization capabilities; improved documentation; etc.
19 
20 GLVis is distributed under the terms of the BSD-3 license. All new contributions
21 must be made under this license.
22 
23 If you plan on contributing to GLVis, consider reviewing the
24 [issue tracker](https://github.com/glvis/glvis/issues) first to check if a thread
25 already exists for your desired feature or the bug you ran into. Use a pull
26 request (PR) toward the `glvis:master` branch to propose your contribution. If
27 you are planning significant code changes or have questions, you may want to
28 open an [issue](https://github.com/glvis/glvis/issues) before issuing a PR. In
29 addition to technical contributions, we are also interested in your results and
30 [simulation images](https://glvis.org/gallery/), which you can share via a pull
31 request in the [glvis/web](https://github.com/glvis/web) repo.
32 
33 See the [Quick Summary](#quick-summary) section for the main highlights of our
34 GitHub workflow. For more details, consult the following sections and refer
35 back to them before issuing pull requests:

60 ## Quick Summary
61 
62 - We encourage you to [join the GLVis organization](#glvis-organization) and create
63  development branches off `glvis:master`.
64 - Please follow the [developer guidelines](#developer-guidelines), in particular
65  with regards to documentation and code styling.
66 - Pull requests should be issued toward `glvis:master`. Make sure
67  to check the items off the [Pull Request Checklist](#pull-request-checklist).
68 - When your contribution is fully working and ready to be reviewed, add
69  the `ready-for-review` label.
70 - PRs are treated similarly to journal submission with an "editor" assigning two
71  reviewers to evaluate the changes.
72 - The reviewers have 3 weeks to evaluate the PR and work with the author to
73  fix issues and implement improvements.
74 - We use [milestones](https://github.com/glvis/glvis/milestones) to coordinate the
75  work on different PRs toward a release.
76 - Don't hesitate to [contact us](#contact-information) if you have any questions.

79 ### Code Overview
80 
81 #### Source code structure:
82 
83 The GLVis library uses object-oriented design principles which reflect, in code,
84 the independent mathematical concepts of meshing, linear algebra and finite
85 element spaces and operators.
86 
87 The GLVis source code has the following structure:
88 
89 ```
90  .
91  ├── lib
92  │ └── gl
93  │ └── shaders
94  ├── share
95  └── tests
96 ```

138 ### New Feature Development
139 
140 - A new feature should be important enough that at least one person, the
141  author, is willing to work on it and be its champion.
142 
143 - The author creates a branch for the new feature (with suffix `-dev`), off
144  the `master` branch, or another existing feature branch, for example:
145 
146  ```
147  # Clone assuming you have setup your ssh keys on GitHub:
148  git clone git@github.com:glvis/glvis.git
149 
150  # Alternatively, clone using the "https" protocol:
151  git clone https://github.com/glvis/glvis.git
152 
153  # Create a new feature branch starting from "master":
154  git checkout master
155  git pull
156  git checkout -b feature-dev
157 
158  # Work on "feature-dev", add local commits
159  # ...
160 
161  # (One time only) push the branch to github and setup your local
162  # branch to track the github branch (for "git pull"):
163  git push -u origin feature-dev
164 
165  ```
166 
167 - **We prefer that you create the new feature branch inside the GLVis organization
168  as opposed to in a fork.** This allows everyone in the community to collaborate
169  in one central place.
170 
171  - If you prefer to work in your fork, please [enable upstream edits](https://help.github.com/articles/allowing-changes-to-a-pull-request-branch-created-from-a-fork/).
172 
173  - Never use the `next` branch to start a new feature branch!
174 
175 - The typical feature branch name is `new-feature-dev`, e.g. `pumi-dev`. While
176  not frequent in GLVis, other suffixes are possible, e.g. `-fix`, `-doc`, etc.

179 ### Developer Guidelines
180 
181 - *Keep the code lean and as simple as possible*
182  - Well-designed simple code is frequently more general and powerful.
183  - Lean code base is easier to understand by new collaborators.
184  - New features should be added only if they are necessary or generally useful.
185  - Introduction of language constructions not currently used in GLVis should be
186  justified and generally avoided (to maintain portability to various systems
187  and compilers, including early access hardware).
188  - We prefer basic C++ and the C++03 standard, to keep the code readable by
189  a large audience and to make sure it compiles anywhere.
190 
191 - *Keep the code general and reasonably efficient*
192  - The main goal is fast prototyping for research and application development.
193  - When in doubt, generality wins over efficiency.
194  - Respect the needs of different users (current and/or future).
195 
196 - *Keep things separate and logically organized*
197  - General usage features go in GLVis (implemented in as much generality as
198  possible), non-general features go into external apps.
199  - Inside GLVis, compartmentalize between linalg, fem, mesh, GLVis, etc.
200  - Contributions that are project-specific or have external dependencies are
201  allowed (if they are of broader interest), but should be `#ifdef`-ed and not
202  change the code by default.
203 
204 - Code specifics
205  - All significant new classes, methods and functions have Doxygen-style
206  documentation in source comments.
207  - Consistent code styling is enforced with `make style` in the top-level
208  directory. This requires [Artistic Style](http://astyle.sourceforge.net)
209  version 3.1 and MFEM's style configuration file, typically located in
210  `../mfem/config/mfem.astylerc`.
211  - When manually resolving conflicts during a merge, make sure to mention the
212  conflicted files in the commit message.

215 ### Pull Requests
216 
217 - When your branch is ready for other developers to review / comment on
218  the code, create a pull request towards `glvis:master`.
219 
220 - Pull request typically have titles like:
221 
222  `Description [new-feature-dev]`
223 
224  for example:
225 
226  `Parallel Unstructured Mesh Infrastructure (PUMI) integration [pumi-dev]`
227 
228  Note the branch name suffix (in square brackets).
229 
230 - Titles may contain a prefix in square brackets to emphasize the type of PR.
231  Common choices are: `[DON'T MERGE]`, `[WIP]` and `[DISCUSS]`, for example:
232 
233  `[DISCUSS] Hybridized DG [hdg-dev]`
234 
235 - If the PR is still a work in progress, add the `WIP` label to it and
236  optionally the `[WIP]` prefix in the title.
237 
238 - Add a description, appropriate labels and assign yourself to the PR. The GLVis
239  team will add reviewers as appropriate.
240 
241 - List outstanding TODO items in the description, see PR #222 for an example.
242 
243 - When your contribution is fully working and ready to be reviewed, add
244  the `ready-for-review` label.
245 
246 - PRs are treated similarly to journal submission with an "editor" assigning
247  two reviewers to evaluate the changes. The reviewers have 3 weeks to evaluate
248  the PR and work with the author to implement improvements and fix issues.
249 
250 ### Pull Request Checklist
251 
252 Before a PR can be merged, it should satisfy the following:
253 
254 - [ ] Code builds.
255 - [ ] Code passes `make style`.
256 - [ ] Update `CHANGELOG`:
257  - [ ] Is this a new feature users need to be aware of?
258  - [ ] Does it make sense to create a new section in the `CHANGELOG` to group with other related features?
259 - [ ] Update `INSTALL`:
260  - [ ] Had a new optional library been added? If so, what range of versions of this library are required? (*Make sure the external library is compatible with our BSD license, e.g. it is not licensed under GPL!*)
261  - [ ] Have the version ranges for any required or optional libraries changed?
262  - [ ] Does `make` or `cmake` have a new target?
263  - [ ] Did the requirements or the installation process change? *(rare)*
264 - [ ] Update `.gitignore`:
265  - [ ] Check if `make distclean; git status` shows any files that were generated from the source by the project (not an IDE) but we don't want to track in the repository.
266  - [ ] Add new patterns (just for the new files above) and re-run the above test.
267 - [ ] New capability:
268  - [ ] All significant new classes, methods and functions have Doxygen-style documentation in source comments.
269  - [ ] Consider saving cool simulation pictures with the new capability in the Confluence gallery (LLNL only) or submitting them, via pull request, to the gallery section of the `glvis/web` repo.
270  - [ ] If this is a major new feature, consider mentioning it in the short summary inside `README` *(rare)*.
271  - [ ] List major new classes in `doc/CodeDocumentation.dox` *(rare)*.
272 - [ ] Update this checklist, if the new pull request affects it.

276 ### Releases
277 
278 - Releases are just tags in the `master` branch, e.g. https://github.com/glvis/glvis/releases/tag/v3.4,
279  and have a version that ends in an even "patch" number, e.g. `v3.4.2` or
280  `v3.4` (by convention `v3.4` is the same as `v3.4.0`.) Between releases, the
281  version ends in an odd "patch" number, e.g. `v3.4.1`.
282 
283 - We use [milestones](https://github.com/glvis/glvis/milestones) to coordinate the
284  work on different PRs toward a release, see for example the
285  [v3.4 release](https://github.com/glvis/glvis/milestone/1?closed=1).
286 
287 - After a release is complete, the `next` branch is recreated, e.g. as follows
288  (replace `3.4` with current release):
289  - Rename the current `next` branch to `next-pre-v3.4`.
290  - Create a new `next` branch starting from the `v3.4` release.
291  - Local copies of `next` can then be updated with `git fetch origin next && git checkout -B next origin/next`.
292 
293 ### Release Checklist
294 - [ ] Update the GLVis version in the following files:
295  - [ ] `CHANGELOG`
296  - [ ] `README.md`
297  - [ ] `vcpkg.json`
298  - [ ] `share/Info.plist`
299  - [ ] `share/Info.cmake.plist.in`
300 - [ ] Check that version requirements for each of GLVis's dependencies are documented in `INSTALL` and up-to-date
301 - [ ] Update the `CHANGELOG` to organize all release contributions
302 - [ ] Review the whole source code once over
303 - [ ] Ask GLVis-based applications to test the pre-release branch
304 - [ ] Test on additional platforms and compilers
305 - [ ] Tag the repository:
306 
307  ```
308  git tag -a v3.1 -m "Official release v3.1"
309  git push origin v3.1
310  ```
311 - [ ] Create the release tarball and push to `glvis/releases`.
312 - [ ] Recreate the `next` branch as described in previous section.
313 - [ ] Update and push documentation to `glvis/doxygen`.
314 - [ ] Update URL shortlinks:
315  - [ ] Create a shortlink at [http://bit.ly/](http://bit.ly/) for the release tarball, e.g. https://glvis.github.io/releases/glvis-3.1.tgz.
316  - [ ] (LLNL only) Add and commit the new shortlink in the `links` and `links-glvis` files of the internal `glvis/downloads` repo.
317  - [ ] Add the new shortlinks to the GLVis packages in `spack`, `homebrew/science`, `VisIt`, etc.
318 - [ ] Update website in `glvis/web` repo:
319  - Update version and shortlinks in `src/index.md` and `src/download.md`.
320  - Use [cloc-1.62.pl](http://cloc.sourceforge.net/) and `ls -lh` to estimate the SLOC and the tarball size in `src/download.md`.

323 ## LLNL Workflow
324 
325 ### Mirroring on Bitbucket
326 
327 - The GitHub `master` and `next` branches are mirrored to the LLNL institutional
328  Bitbucket repository as `gh-master` and `gh-next`.
329 
330 - `gh-master` is merged into LLNL's internal `master` through pull requests; write
331  permissions to `master` are restricted to ensure this is the only way in which it
332  gets updated.
333 
334 - We never push directly from LLNL to GitHub.
335 
336 - Versions of the code on LLNL's internal server, from most to least stable:
337  - GLVis official release on glvis.org -- Most stable, tested in many apps.
338  - `glvis:master` -- Recent development version, guaranteed to work.
339  - `glvis:gh-master` -- Stable development version, passed testing, you can use
340  it to build your code between releases.
341  - `glvis:gh-next` -- Bleeding-edge development version, may be broken, use at
342  your own risk.
